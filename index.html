Git কী?
ভার্সন কন্ট্রোলের জগতে স্বাগতম

Git হলো একটি ডিস্ট্রিবিউটেড ভার্সন কন্ট্রোল সিস্টেম। সহজ ভাষায়, এটি তোমার কোডের সব পরিবর্তনের ইতিহাস রাখে — কখন কী পরিবর্তন হয়েছে, কে করেছে, কেন করেছে — সব কিছু ট্র্যাক করে।


Git কেন দরকার? ধরো তুমি একটা ফিচার বানাতে গিয়ে পুরো প্রোজেক্ট নষ্ট করে ফেললে — Git দিয়ে এক কমান্ডেই আগের অবস্থায় ফিরে যেতে পারবে। দলে কাজ করার সময় একসাথে একই ফাইলে কাজ করা যায়। GitHub/GitLab-এ কোড হোস্ট করা যায়।


Git ব্যবহার করে তুমি মূলত ৪টি এলাকায় কাজ করো: Working Directory (তোমার ফাইল), Staging Area (commit-এর জন্য প্রস্তুত ফাইল), Local Repository (তোমার মেশিনে সেভ করা ইতিহাস), এবং Remote Repository (GitHub-এ থাকা ক্লাউড কপি)।

🚀
শুরু করার কমান্ড
নতুন রেপো তৈরি বা ক্লোন করা
INIT
git init
নতুন Git রেপো তৈরি করো
🔷 কী করে
বর্তমান ফোল্ডারকে একটি Git রেপোজিটরিতে পরিণত করে। একটি লুকানো .git ফোল্ডার তৈরি করে যেখানে সব ইতিহাস সংরক্ষিত হয়।
🔶 কেন ব্যবহার করবে
যখন নতুন একটি প্রোজেক্ট শুরু করছ এবং সেটার ভার্সন ট্র্যাক করতে চাও।
🕐 কখন
প্রোজেক্টের একদম শুরুতে, একবারই ব্যবহার করতে হয়।
উদাহরণ
mkdir my-project && cd my-project git init # এই ফোল্ডারে Git শুরু করো git init my-folder # নতুন ফোল্ডার তৈরি করে init
INIT
git clone <url>
দূরের রেপো কপি করে আনো
🔷 কী করে
GitHub বা অন্য কোনো সার্ভার থেকে একটি রেপোজিটরির সম্পূর্ণ কপি তোমার মেশিনে নিয়ে আসে — সব ইতিহাস সহ।
🔶 কেন ব্যবহার করবে
কোনো open-source প্রোজেক্টে contribute করতে বা টিমের প্রোজেক্টে যোগ দিতে।
🕐 কখন
যখন প্রথমবার কোনো রেমোট প্রোজেক্ট লোকালে নামাতে হবে।
উদাহরণ
git clone https://github.com/user/repo.git git clone <url> my-folder # নির্দিষ্ট ফোল্ডারে git clone --depth 1 <url> # শুধু শেষ commit
INIT
git config
Git সেটিংস কনফিগার করো
🔷 কী করে
তোমার নাম, ইমেইল, এডিটর ইত্যাদি Git-এর কনফিগারেশন সেট করে। এই তথ্য প্রতিটি commit-এ যুক্ত হয়।
🔶 কেন ব্যবহার করবে
Git ইনস্টলের পরপরই পরিচয় সেট করা দরকার। ছাড়া commit হবে না।
🕐 কখন
Git ইনস্টলের প্রথম দিনেই করতে হয়।
উদাহরণ
git config --global user.name "তোমার নাম" git config --global user.email "email@example.com" git config --list # সব সেটিং দেখো
📁
মূল কমান্ড
রোজকার কাজে যা সবচেয়ে বেশি লাগে
BASIC
git status
রেপোর বর্তমান অবস্থা দেখো
🔷 কী করে
কোন ফাইল পরিবর্তন হয়েছে, কোনটি staged আছে, কোনটি untracked — এই সব তথ্য দেখায়।
🔶 কেন ব্যবহার করবে
Commit করার আগে সব ঠিক আছে কিনা দেখতে। এটি সবচেয়ে বেশি ব্যবহৃত কমান্ড।
🕐 কখন
যেকোনো কাজের আগে-পরে। যখনই সন্দেহ হবে।
উদাহরণ
git status git status -s # সংক্ষিপ্ত আউটপুট
BASIC
git add
ফাইল staging area-তে রাখো
🔷 কী করে
নির্দিষ্ট ফাইলগুলোকে staging area-তে যোগ করে, মানে পরবর্তী commit-এ অন্তর্ভুক্ত করার জন্য চিহ্নিত করে।
🔶 কেন ব্যবহার করবে
শুধু নির্দিষ্ট পরিবর্তনগুলো commit করতে চাইলে। সব ফাইল একবারে না করে আলাদা করা যায়।
🕐 কখন
Commit করার ঠিক আগে।
উদাহরণ
git add index.html # একটি ফাইল git add . # সব পরিবর্তিত ফাইল git add *.js # সব .js ফাইল git add -p # অংশ বিশেষ করে add
BASIC
git commit
পরিবর্তন স্থায়ীভাবে সেভ করো
🔷 কী করে
Staging area-তে থাকা পরিবর্তনগুলো একটি snapshot হিসেবে রেপোতে সেভ করে। প্রতিটি commit-এর একটি অনন্য ID (hash) থাকে।
🔶 কেন ব্যবহার করবে
কাজের একটি milestone ধরে রাখতে। ভবিষ্যতে ফিরে আসা যাবে।
🕐 কখন
একটি ফিচার বা ফিক্স সম্পন্ন হলে। ছোট ছোট commit করা ভালো অভ্যাস।
উদাহরণ
git commit -m "Login feature যোগ করা হয়েছে" git commit -am "Bug fix" # add + commit একসাথে git commit --amend # শেষ commit সংশোধন
BASIC
git log
commit ইতিহাস দেখো
🔷 কী করে
সব commit-এর তালিকা দেখায় — তারিখ, লেখক, বার্তা সহ। নতুন commit সবার উপরে থাকে।
🔶 কেন ব্যবহার করবে
কে কখন কী পরিবর্তন করেছে জানতে। নির্দিষ্ট commit খুঁজতে।
🕐 কখন
ইতিহাস পর্যালোচনার সময়, bug খুঁজতে, বা পুরনো version-এ ফিরতে।
উদাহরণ
git log git log --oneline # সংক্ষিপ্ত তালিকা git log --graph --all # ব্রাঞ্চ গ্রাফ git log -5 # শেষ ৫টি commit
BASIC
git diff
পরিবর্তনের বিস্তারিত দেখো
🔷 কী করে
ঠিক কোন লাইনে কী পরিবর্তন হয়েছে সেটা line-by-line দেখায়। সবুজ = যোগ করা, লাল = মুছা।
🔶 কেন ব্যবহার করবে
Commit করার আগে কী পরিবর্তন করেছ review করতে।
🕐 কখন
git add করার আগে পরিবর্তন যাচাই করতে।
উদাহরণ
git diff # unstaged পরিবর্তন git diff --staged # staged পরিবর্তন git diff main feature # দুই ব্রাঞ্চ তুলনা
BASIC
git show
নির্দিষ্ট commit-এর বিবরণ
🔷 কী করে
একটি নির্দিষ্ট commit-এ কী পরিবর্তন হয়েছিল সেটা বিস্তারিত দেখায়।
🔶 কেন ব্যবহার করবে
কোনো পুরনো commit কী করেছিল জানতে হলে।
🕐 কখন
Bug tracing বা code review-এর সময়।
উদাহরণ
git show # শেষ commit দেখো git show abc1234 # নির্দিষ্ট commit git show HEAD~2 # ২টি আগের commit
🌿
ব্রাঞ্চ ও মার্জ
আলাদা লাইনে কাজ করো, তারপর একত্রিত করো
BRANCH
git branch
ব্রাঞ্চ তৈরি ও দেখো
🔷 কী করে
ব্রাঞ্চের তালিকা দেখায়, নতুন ব্রাঞ্চ তৈরি করে, বা মুছে দেয়। ব্রাঞ্চ হলো মূল কোড থেকে আলাদা একটি copy যেখানে স্বাধীনভাবে কাজ করা যায়।
🔶 কেন ব্যবহার করবে
নতুন feature বানাতে বা bug fix করতে main code না ভেঙে আলাদা জায়গায় কাজ করতে।
🕐 কখন
যেকোনো নতুন কাজ শুরু করার আগে।
উদাহরণ
git branch # সব ব্রাঞ্চ দেখো git branch login-feature # নতুন ব্রাঞ্চ তৈরি git branch -d login-feature # ব্রাঞ্চ মুছো git branch -a # remote সহ সব ব্রাঞ্চ
BRANCH
git checkout
ব্রাঞ্চ বদলাও বা ফাইল ফেরাও
🔷 কী করে
এক ব্রাঞ্চ থেকে অন্য ব্রাঞ্চে সরে যায়। অথবা কোনো ফাইলকে আগের অবস্থায় ফিরিয়ে দেয়।
🔶 কেন ব্যবহার করবে
ভিন্ন feature-এ কাজ করতে বা পুরনো কোড দেখতে।
🕐 কখন
যখন ব্রাঞ্চ বদলাতে হবে।
উদাহরণ
git checkout main # main ব্রাঞ্চে যাও git checkout -b new-feature # বানাও ও চলে যাও git checkout -- index.html # ফাইল ফিরিয়ে আনো
BRANCH
git switch
ব্রাঞ্চ পরিবর্তন (নতুন পদ্ধতি)
🔷 কী করে
git checkout-এর মতোই কাজ করে তবে শুধু ব্রাঞ্চ পরিবর্তনের জন্য। আধুনিক Git-এ এটাই প্রস্তাবিত।
🔶 কেন ব্যবহার করবে
checkout-এর চেয়ে বেশি স্পষ্ট, কম বিভ্রান্তিকর।
🕐 কখন
ব্রাঞ্চ পরিবর্তনের যেকোনো সময়।
উদাহরণ
git switch main git switch -c new-feature # তৈরি করে চলে যাও git switch - # আগের ব্রাঞ্চে ফিরে যাও
BRANCH
git merge
দুটি ব্রাঞ্চ একত্রিত করো
🔷 কী করে
একটি ব্রাঞ্চের সব পরিবর্তন বর্তমান ব্রাঞ্চে নিয়ে আসে এবং একটি merge commit তৈরি করে।
🔶 কেন ব্যবহার করবে
Feature development শেষে main ব্রাঞ্চে যুক্ত করতে।
🕐 কখন
Feature বা bug-fix ব্রাঞ্চ সম্পন্ন হলে।
উদাহরণ
git switch main git merge feature-login # merge করো git merge --no-ff feature # merge commit জোর করো git merge --abort # conflict হলে বাতিল
BRANCH
git rebase
commit ইতিহাস পুনর্লিখন
🔷 কী করে
একটি ব্রাঞ্চের commit গুলো অন্য ব্রাঞ্চের উপর নতুনভাবে প্রয়োগ করে। ইতিহাস সরলরেখায় থাকে।
🔶 কেন ব্যবহার করবে
পরিষ্কার, linear ইতিহাস রাখতে চাইলে merge-এর বদলে rebase ব্যবহার করা হয়।
🕐 কখন
Push করার আগে local commit পরিষ্কার করতে। ⚠️ Shared branch-এ rebase করবে না।
উদাহরণ
git rebase main # main-এর উপর rebase git rebase -i HEAD~3 # শেষ ৩ commit edit git rebase --abort # বাতিল করো
BRANCH
git stash
কাজ সাময়িক লুকিয়ে রাখো
🔷 কী করে
Uncommitted পরিবর্তনগুলো একটি অস্থায়ী জায়গায় সরিয়ে রাখে, working directory পরিষ্কার করে।
🔶 কেন ব্যবহার করবে
মাঝপথে ব্রাঞ্চ বদলাতে হলে কিন্তু commit করার মতো হয়নি।
🕐 কখন
জরুরি bug fix করতে অন্য ব্রাঞ্চে যেতে হলে।
উদাহরণ
git stash # কাজ লুকাও git stash pop # ফিরিয়ে আনো git stash list # সব stash দেখো git stash apply stash@{2}
☁️
রিমোট রেপোজিটরি
GitHub, GitLab বা অন্য সার্ভারের সাথে কাজ
REMOTE
git remote
রিমোট সংযোগ ব্যবস্থাপনা
🔷 কী করে
Remote repository-র সংযোগ দেখায়, যোগ করে বা সরায়। সাধারণত origin নামে GitHub-এর লিংক থাকে।
🔶 কেন ব্যবহার করবে
কোথায় push বা pull হচ্ছে সেটা সেট বা পরিবর্তন করতে।
🕐 কখন
Local repo-কে GitHub-এর সাথে সংযুক্ত করার সময়।
উদাহরণ
git remote -v # সংযোগ দেখো git remote add origin <url> git remote remove origin git remote rename origin upstream
REMOTE
git push
রিমোটে কোড আপলোড করো
🔷 কী করে
Local repository-র commit গুলো remote (যেমন GitHub) এ আপলোড করে।
🔶 কেন ব্যবহার করবে
কোড backup রাখতে, টিমের সাথে শেয়ার করতে।
🕐 কখন
Commit শেষ হলে, অন্যদের সাথে শেয়ার করতে চাইলে।
উদাহরণ
git push origin main git push -u origin feature # প্রথমবার git push --all # সব ব্রাঞ্চ push git push --force-with-lease # সতর্কে force push
REMOTE
git pull
রিমোট থেকে আপডেট আনো
🔷 কী করে
Remote থেকে নতুন পরিবর্তন নিয়ে আসে এবং সাথে সাথে merge করে দেয়। এটি আসলে fetch + merge একসাথে।
🔶 কেন ব্যবহার করবে
টিমের অন্যরা যে পরিবর্তন করেছে সেটা নিজের কাছে আনতে।
🕐 কখন
কাজ শুরু করার আগে সবসময়। Push করার আগে।
উদাহরণ
git pull origin main git pull --rebase # rebase দিয়ে pull git pull # default remote থেকে
REMOTE
git fetch
রিমোট info আনো, merge নয়
🔷 কী করে
Remote থেকে নতুন পরিবর্তন ডাউনলোড করে কিন্তু local ব্রাঞ্চে merge করে না। দেখার সুযোগ দেয়।
🔶 কেন ব্যবহার করবে
কী পরিবর্তন আসছে আগে দেখতে চাইলে, merge না করেই।
🕐 কখন
Remote এ কী হয়েছে review করতে চাইলে।
উদাহরণ
git fetch origin git fetch --all # সব remote থেকে git diff main origin/main # পার্থক্য দেখো
↩️
পূর্বাবস্থায় ফেরা
ভুল হলে ফিরে যাও — Git সবসময় সাহায্য করবে
UNDO
git revert
নিরাপদে commit বাতিল করো
🔷 কী করে
একটি commit-এর বিপরীত কাজ করে নতুন একটি commit তৈরি করে। পুরনো ইতিহাস অক্ষত থাকে।
🔶 কেন ব্যবহার করবে
Push করা commit বাতিল করতে হলে এটাই সঠিক পদ্ধতি। ইতিহাস নষ্ট হয় না।
🕐 কখন
Shared/public branch-এ ভুল commit ঠিক করতে।
উদাহরণ
git revert HEAD # শেষ commit বাতিল git revert abc1234 # নির্দিষ্ট commit git revert HEAD~3..HEAD # শেষ ৩টি
UNDO
git reset
commit পয়েন্টার সরাও
🔷 কী করে
HEAD pointer-কে পূর্ববর্তী commit-এ নিয়ে যায়। --soft (ফাইল রাখে), --mixed (unstaged করে), --hard (সব মুছে)।
🔶 কেন ব্যবহার করবে
Push করার আগে local commit ঠিক করতে।
🕐 কখন
⚠️ শুধু local ব্রাঞ্চে। Push করা commit-এ reset করবে না।
উদাহরণ
git reset --soft HEAD~1 # commit বাতিল, ফাইল staged git reset HEAD~1 # unstaged করে রাখে git reset --hard HEAD~1 # ⚠️ সব মুছে যাবে
UNDO
git restore
ফাইল পুনরুদ্ধার করো
🔷 কী করে
Working directory বা staging area-র ফাইলকে পূর্ববর্তী অবস্থায় ফিরিয়ে আনে।
🔶 কেন ব্যবহার করবে
ভুলভাবে কোনো ফাইল পরিবর্তন করে ফেললে।
🕐 কখন
Commit করার আগে কোনো পরিবর্তন বাতিল করতে।
উদাহরণ
git restore index.html # ফাইল ফেরাও git restore --staged file.js # unstage করো git restore . # সব ফাইল ফেরাও
UNDO
git clean
Untracked ফাইল মুছো
🔷 কী করে
Git-এ track করা হয়নি এমন ফাইল এবং ফোল্ডার মুছে দেয়।
🔶 কেন ব্যবহার করবে
Build files বা অপ্রয়োজনীয় ফাইল দ্রুত পরিষ্কার করতে।
🕐 কখন
Directory পরিষ্কার করতে চাইলে।
উদাহরণ
git clean -n # কী মুছবে দেখো (dry run) git clean -f # ফাইল মুছো git clean -fd # ফাইল + ফোল্ডার
⚡
অ্যাডভান্সড কমান্ড
দক্ষ developer-দের টুলস
ADV
git cherry-pick
নির্দিষ্ট commit তুলে আনো
🔷 কী করে
অন্য ব্রাঞ্চ থেকে একটি নির্দিষ্ট commit বর্তমান ব্রাঞ্চে প্রয়োগ করে, পুরো ব্রাঞ্চ merge না করে।
🔶 কেন ব্যবহার করবে
অন্য ব্রাঞ্চের একটি নির্দিষ্ট bug fix বা feature শুধু নিজের ব্রাঞ্চে আনতে।
🕐 কখন
Hotfix একাধিক ব্রাঞ্চে প্রয়োগ করতে।
উদাহরণ
git cherry-pick abc1234 git cherry-pick abc1234 def5678 # একাধিক git cherry-pick main~3..main
ADV
git tag
ভার্সন চিহ্নিত করো
🔷 কী করে
একটি নির্দিষ্ট commit-কে একটি নাম দেয় (যেমন v1.0.0)। Release বা milestone চিহ্নিত করতে ব্যবহার হয়।
🔶 কেন ব্যবহার করবে
Software release ট্র্যাক করতে, ভার্সন ম্যানেজমেন্টে।
🕐 কখন
নতুন version release করার সময়।
উদাহরণ
git tag v1.0.0 git tag -a v1.0.0 -m "প্রথম রিলিজ" git push origin --tags git tag -l # সব tag দেখো
ADV
git bisect
Bug-এর উৎস খুঁজে বের করো
🔷 কী করে
Binary search পদ্ধতিতে commit ইতিহাস ঘেঁটে কোন commit-এ bug ঢুকেছে সেটা বের করে।
🔶 কেন ব্যবহার করবে
হাজারো commit-এর মধ্যে দ্রুত bug-এর উৎস খুঁজতে।
🕐 কখন
কখন bug ঢুকেছে জানা নেই কিন্তু কোনো commit-এ ঠিক ছিল।
উদাহরণ
git bisect start git bisect bad # এখন bug আছে git bisect good v1.0 # এখানে ভালো ছিল git bisect reset # শেষ করো
ADV
git blame
কোন লাইন কে লিখেছে জানো
🔷 কী করে
একটি ফাইলের প্রতিটি লাইনে কে, কখন, কোন commit-এ লিখেছে সেটা দেখায়।
🔶 কেন ব্যবহার করবে
কোনো code কে লিখেছে বা কোন commit-এ এই লাইন এসেছে জানতে।
🕐 কখন
Code review বা bug tracing-এর সময়।
উদাহরণ
git blame index.html git blame -L 10,20 app.js # ১০-২০ নং লাইন
ADV
git reflog
HEAD-এর সব নড়াচড়া দেখো
🔷 কী করে
HEAD pointer কখন কোথায় ছিল তার সম্পূর্ণ ইতিহাস রাখে। হারানো commit পুনরুদ্ধারের শেষ অস্ত্র।
🔶 কেন ব্যবহার করবে
git reset --hard বা ভুল rebase-এ হারানো কাজ ফিরে পেতে।
🕐 কখন
⚠️ জরুরি অবস্থায়, কোড হারিয়ে গেলে।
উদাহরণ
git reflog git checkout HEAD@{3} # ৩ steps আগে যাও git reset --hard HEAD@{2}
ADV
git submodule
রেপোর মধ্যে রেপো রাখো
🔷 কী করে
একটি রেপোকে অন্য রেপোর ভেতরে একটি নির্দিষ্ট commit-এ link করে রাখে।
🔶 কেন ব্যবহার করবে
বড় প্রোজেক্টে external dependency বা shared library আলাদা রেপো হিসেবে রাখতে।
🕐 কখন
Monorepo setup বা external library ট্র্যাক করতে।
উদাহরণ
git submodule add <url> libs/name git submodule update --init git submodule update --recursive

গুরুত্বপূর্ণ টিপস
ভুল করার আগেই জানো
✅
ছোট ছোট commit করো
একটি কাজ শেষ হলেই commit করো। বড় commit debug করা কঠিন এবং ইতিহাস বোঝা যায় না।
📝
ভালো commit message লেখো
"fix bug" নয়, লেখো "লগইন পেজে পাসওয়ার্ড validation ঠিক করা হয়েছে"। ভবিষ্যতে নিজেই বুঝতে পারবে।
⚠️
main ব্রাঞ্চে সরাসরি কাজ নয়
সবসময় নতুন ব্রাঞ্চ খুলে কাজ করো। main বা master ব্রাঞ্চ সবসময় stable রাখো।
🚫
--force push এড়িয়ে চলো
Shared branch-এ git push --force করলে টিমের সবার কাজ মুছে যেতে পারে। অত্যন্ত বিপজ্জনক।
🔒
.gitignore ব্যবহার করো
node_modules/, .env, build/ ইত্যাদি ফাইল track করার দরকার নেই। .gitignore ফাইলে লিখে রাখো।
🔄
কাজ শুরুর আগে pull করো
প্রতিদিন কাজ শুরু করার আগে git pull করো। এতে conflict কম হয় এবং টিমের সাথে sync থাকো।
💾
reset --hard এর আগে ভাবো
git reset --hard করলে সব অপরিবর্তিত কাজ চিরতরে হারাবে। এর আগে stash বা backup নাও।
🔍
reflog তোমার বন্ধু
কোনো কাজ হারিয়ে গেলে হতাশ হয়ো না। git reflog দিয়ে প্রায় যেকোনো কিছু ফিরে পাওয়া যায়।
